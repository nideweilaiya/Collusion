# Collusion 方案管理与知识库体系

> 核心理念：方案不只是正式计划，一切有目的的决策都是方案。
> 知识库不是答案来源，而是风险来源。让模型自由创造，用知识库验证约束。

---

## 一、方案的新定义

### 1.1 什么是"方案"？

在传统 AI 辅助开发中，"方案"特指通过正式命令（如 `/collusion design`）生成的完整架构文档。
但实际项目中大量有价值经验诞生于非正式时刻：一次 Bug 修复、一个被否决的技术提议、一次临时配置调整。

因此，方案的定义被重新界定：**一切有目的的决策过程，只要包含"目标—行动—结果"三要素，就是一个可归档的经验单元。**

### 1.2 方案的六种类型

| 类型 | 识别标准 | 典型场景 |
|------|---------|---------|
| **正式计划** | 通过编排流程生成的完整方案，含多环节设计、技术选型、评分 | 用户主动调用 `brainstorm_orchestrate` |
| **临时修复** | 对话中出现"修复""bug""修好了"，且代码被修改 | 修复"砍树中途挖矿"Bug |
| **默认提案** | Agent 主动给出可选方案，用户选择其他方向 | Agent 建议 Redis，用户决定 SQLite |
| **一次性决策** | "决定""我们就用""确定用"等确认性语言 | "决定使用 Baritone 作为导航引擎" |
| **审查与复盘** | 审查阶段发现的遗漏/风险，或用户说"好了""可以了" | 安全专家发现缺少输入验证 |
| **废弃方案** | 用户明确放弃的方向，或被可行性收束淘汰 | 因过度设计被否决的 PG+Redis 方案 |

## 二、经验分级 L0-L3

| 等级 | 名称 | 定义 | 归档时机 | 存储位置 |
|:----:|------|------|---------|---------|
| **L3** | 已验证经验 | 编译通过 + 测试通过 + 人工确认 | 用户说"好了"后立即归档 | `asset_library/` |
| **L2** | 待验证修改 | 编译通过，待人工确认 | 暂存 24h，确认→L3，否则→L1 | `asset_library/`（待验证） |
| **L1** | 尝试性修改 | 编译未通过或中间态 | 仅保留最后一轮失败原因 | `causal_memory/` |
| **L0** | 废弃片段 | 被明确放弃的方向 | 提取废案元数据后丢弃 | `wiki/pitfalls/` |

### 归档触发（Hook 驱动）

| 时机 | 事件 | 等级 |
|------|------|:----:|
| 任务完成 | `Stop` | L2 |
| 用户确认 | `UserPromptSubmit` | L2→L3 |
| 关键行动 | `PostToolUse` | L1/L2 |
| 定时归档 | 每 30min | L2→L3/L1 |
| 对话中断 | 30min 无交互 | L1 |

### 中断恢复

- 主动中断 → 存 `data/interrupts/`，标记"待续"
- 被动中断 → 从 GoalRunner 持久化恢复
- 静默中断 → 30min 超时自动中断归档
- 恢复：新会话检查中断摘要，<72h 可续，>72h 归档为 L1

## 三、三层知识库架构

```
┌──────────────────────────────────────┐
│  编译层（Wiki）—— Markdown + 双链     │
│  LLM 按 Schema 自动维护、去重、关联    │
├──────────────────────────────────────┤
│  资产层（Asset Library）—— JSON 索引   │
│  GoalRunner 自动归档，向量检索         │
├──────────────────────────────────────┤
│  原始资料层（Raw）——只读，绝不修改     │
│  手动导入，LLM 只读                     │
└──────────────────────────────────────┘
```

## 四、生成-验证双层架构

```
生成器（DeepSeek，不加知识库）→ 自由推理、设计方案
         ↓
验证器（确定性程序 + 知识库审查）→ 编译、测试、标记危险
```

**知识库在不同环节的角色：**

| 环节 | 角色 |
|------|------|
| 方案设计 | 参考案例，不限制方向 |
| 代码生成 | **不介入** |
| 自动验证 | 参考标准 |
| 审查与归档 | **约束守卫** |
| 新知识沉淀 | 更新目标 |

**三种使用模式：** 自由生成 / 参考生成 / 严格约束

## 五、知识复利机制

- **结构化存储**：goal / root_cause / solution / tags / reuse_count
- **复利触发**：相似任务检索 → 因果链复用 → 强度递增 → 衰减不删除
- **跨领域发现**：自动提取实体、概念、关系，发现隐藏关联

---

## 六、废案休眠唤醒

废案记录：废弃原因 + 激活条件 + 跨领域潜力

新任务 → 语义检索包括废案 → 检查激活条件 → 若满足则并排评估 → 重新采纳则升级 L3

---

## 七、健康维护

- 每 10 次归档：矛盾扫描 + 孤立页面清理
- 每 30 天：过时检测
- 调用量驱动：高频完整 → 低频摘要 → 休眠保留摘要（永不删除）
